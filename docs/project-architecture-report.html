3. 数据怎么存、存在哪里

数据类型存放位置存什么我的判断
登录用户 PostgreSQL 表 users 飞书 open_id、姓名、邮箱、头像。 合理 用户基础信息放数据库正确。
登录会话 PostgreSQL 表 auth_sessions 浏览器 cookie 对应的 session token，以及飞书 access token / refresh token。 需保护 这是敏感数据，建议加密或至少强化备份和权限。
用户保存的模板配置 PostgreSQL 表 saved_configs 某个用户在某张表里怎么绑定模板变量、文件名、协作者等配置，存在 payload_json。 当前可用 但 JSON 字段多后可考虑拆出常查字段。
原始业务记录 飞书多维表格 客户名、金额、日期、附件图片等真正的业务数据。 合理 插件不复制整张业务表，只在生成时读取。
Docx 模板资产 TOS 对象存储，生产必须配置 templates/_index.json、metadata.json、versions/v001/source.docx。 方向正确 大文件放对象存储，不塞数据库。
最终生成的 Docx TOS 或 OSS 对象存储 按记录生成出的 Word 文件，返回有有效期的下载链接。 方向正确 生产不应落本机临时目录。
异步任务进度 当前 Node 进程内存 任务 id、进度、结果列表。 短期可用 重启会丢，后续应持久化。

4. 数据表结构

PostgreSQL：真实数据库表

表字段主键 / 约束人话解释
users open_id, name, en_name, email, avatar_url, created_at, updated_at open_id 主键 每个飞书用户一本“用户档案”。
auth_sessions token, oauth_app_key, open_id, access_token, refresh_token, token_type, expires_at, refresh_expires_at, created_at, updated_at token 主键；open_id 外键到 users；有 open_id 和 expires_at 索引 浏览器 cookie 对应的“入场券”，里面还保存飞书访问令牌。
saved_configs id, open_id, config_name, payload_json, created_at, updated_at id 主键；open_id 外键；UNIQUE(open_id, config_name)；有 open_id 和 updated_at 索引 用户保存过的模板绑定方案。比如“这个模板的客户名称绑定到表格的客户列”。

对象存储：不是数据库表，但也是核心数据

对象典型路径内容
模板索引 templates/_index.json 模板列表：templateId、名称、状态、当前版本、变量、缩略图、时间。
模板详情 templates/{templateId}/metadata.json 模板完整记录：版本列表、变量、缩略图、创建人、删除状态等。
模板源文件 templates/{templateId}/versions/v001/source.docx 真正的 Word 模板文件。
生成文件 renders/YYYY/MM/DD/{requestId}.docx 已经替换变量的成品 Word 文件。

可选飞书多维表格同步表

如果启用 BITABLE_SYNC_ENABLED，后端会把配置同步到一张飞书多维表格。字段包括：

user_open_id table_id config_name template_id template_title template_url payload_json created_at updated_at
